Техническое задание
на разработку модуля аппроксимации для GUI-приложения
1. Наименование модуля

Модуль аппроксимации и построения поверхности по экспериментальным данным.

2. Назначение модуля

Модуль предназначен для загрузки исходных табличных данных из CSV-файлов, выполнения аппроксимации экспериментальных точек по двум независимым переменным (fuel, additive) и одной зависимой переменной (component), последующей фильтрации полученной поверхности и сохранения результата в выходной CSV-файл.

Модуль должен быть встроен в графическое приложение и предоставлять пользователю удобный интерфейс для:

выбора входных файлов или папки;
задания параметров аппроксимации;
запуска обработки;
просмотра статуса выполнения;
сохранения результатов.

Текущая вычислительная схема состоит из последовательных этапов: чтение CSV → аппроксимация RBF → фильтрация поверхности → экспорт результата.

3. Основание для разработки

Основанием для разработки является необходимость переноса существующей библиотечной/скриптовой реализации в GUI-модуль с пользовательским управлением параметрами и файловыми операциями.

В существующей реализации предусмотрен пакетный режим обработки: для каждого CSV-файла из входной папки формируется файл approx_<исходное_имя>.csv в выходной папке.

4. Функции модуля

Модуль должен реализовывать следующие функции:

4.1. Загрузка входных данных

Модуль должен обеспечивать:

выбор одного CSV-файла;
выбор нескольких CSV-файлов;
выбор папки с CSV-файлами для пакетной обработки.
4.2. Проверка входных данных

При загрузке каждого CSV-файла модуль должен:

проверять доступность файла для чтения;
считывать CSV с разделителем ;;
проверять наличие обязательных столбцов:
fuel
additive
component.

При отсутствии хотя бы одного обязательного столбца модуль должен выдавать понятное сообщение об ошибке и не запускать обработку данного файла. Это соответствует текущей логике read_csv.

4.3. Аппроксимация данных

Модуль должен выполнять аппроксимацию на основе радиально-базисной интерполяции/аппроксимации (RBFInterpolator) по точкам вида:

X = fuel
Y = additive
Z = component.

На основе диапазонов значений fuel и additive должна строиться равномерная сетка заданного разрешения (Nx, Ny), после чего на этой сетке рассчитываются значения поверхности. Это соответствует реализованному алгоритму rbf_approx.

4.4. Фильтрация результата

После построения поверхности модуль должен обеспечивать постобработку:

замену отрицательных значений на 0 при включенном режиме clamp_zero;
применение медианного фильтра с параметром median_size.

Фильтрация должна выполняться в указанной последовательности.

4.5. Экспорт результата

Результирующая поверхность должна сохраняться в CSV-файл в плоском табличном виде со столбцами:

fuel
additive
component.

Запись должна выполняться без индекса DataFrame. Формат сохранения должен соответствовать существующей логике export_surface.

4.6. Пакетная обработка

Модуль должен поддерживать обработку всех CSV-файлов в выбранной папке. Для каждого входного файла должен создаваться отдельный выходной файл.

Имена выходных файлов по умолчанию должны формироваться по шаблону:
approx_<имя_входного_файла>.csv.

5. Требования к пользовательскому интерфейсу
5.1. Общие требования

Интерфейс должен быть реализован как отдельная вкладка или отдельное окно GUI-приложения.

В окне модуля должны присутствовать следующие элементы:

Блок выбора источника данных
поле для отображения пути к входному файлу;
кнопка «Выбрать файл»;
поле для отображения пути к входной папке;
кнопка «Выбрать папку»;
переключатель режима:
одиночный файл;
пакетная обработка папки.
Блок выбора пути сохранения
поле для отображения выходного файла или папки;
кнопка выбора выходной директории;
опция автогенерации имени выходного файла.
Блок параметров аппроксимации

Пользователь должен иметь возможность задавать:

resolution_x — количество узлов по оси fuel;
resolution_y — количество узлов по оси additive;
kernel — тип ядра RBF;
median_size — размер окна медианного фильтра;
clamp_zero — флаг обнуления отрицательных значений.
Блок управления
кнопка «Проверить данные»;
кнопка «Запустить обработку»;
кнопка «Остановить» — опционально, если обработка будет выполняться асинхронно;
кнопка «Открыть результат» или «Открыть папку результата».
Блок вывода информации
текстовый лог выполнения;
индикатор прогресса;
статусная строка.
5.2. Требования к удобству использования

Интерфейс должен:

блокировать запуск обработки при отсутствии корректного входного пути;
блокировать запуск при незаполненном выходном пути;
выполнять базовую валидацию параметров до старта;
отображать понятные сообщения об ошибках на русском языке;
отображать успешное завершение обработки;
в пакетном режиме показывать, какой файл обрабатывается в текущий момент.
6. Функциональные требования
6.1. Требования к входным данным

Входной CSV-файл должен:

быть текстовым CSV-файлом;
использовать разделитель ;;
содержать заголовок;
содержать столбцы fuel, additive, component.

Значения в этих столбцах должны интерпретироваться как числовые.

6.2. Требования к параметрам аппроксимации
Разрешение сетки

Параметр resolution должен задаваться как пара целых положительных чисел:

resolution_x > 0
resolution_y > 0.

По умолчанию может использоваться значение (100, 100), как в текущем скрипте.

Размер медианного фильтра

Параметр median_size должен быть:

целым числом;
неотрицательным;
при значении 0 или пустом значении фильтр может не применяться.
Текущая логика предполагает применение фильтра только если параметр задан.
Флаг clamp_zero

Параметр должен задаваться как булев:

True — отрицательные значения заменяются на 0;
False — значения сохраняются без ограничения.
Тип ядра

Параметр kernel должен задаваться строковым значением.
По умолчанию — "linear".

Желательно предусмотреть выбор из списка допустимых значений, поддерживаемых используемой библиотекой интерполяции.

6.3. Требования к обработке

При запуске обработки для одного файла модуль должен:

Считать CSV.
Проверить состав столбцов.
Построить поверхность по RBF.
Выполнить фильтрацию.
Сохранить результат.
Вывести сообщение об успехе либо ошибке.

При запуске пакетной обработки модуль должен:

Просканировать выбранную папку.
Выбрать все файлы с расширением .csv.
Последовательно выполнить обработку для каждого файла.
Создать выходную папку при её отсутствии.
Продолжать обработку остальных файлов даже при ошибке в одном из них.
Вести журнал ошибок по каждому файлу.
Именно такой принцип реализован в текущем main.py.
Требования к входным и выходным CSV-файлам
1. Назначение входного CSV-файла

На вход модуля поступает CSV-файл, содержащий результаты газового анализа, а именно значения концентрации исследуемого компонента.

Каждый CSV-файл представляет собой набор экспериментальных данных, по которым далее выполняется аппроксимация поверхности.

2. Требования к имени входного файла

Имя входного CSV-файла должно иметь следующий формат:

наименование_топлива-разбавитель-компонент-дд-мм-гггг.csv

Пример:
waste_oil-steam-CO-13-03-2026.csv

Расшифровка элементов имени файла:
наименование_топлива — идентификатор исследуемого топлива;
разбавитель — используемый разбавитель;
компонент — исследуемый компонент, концентрация которого измерялась;
дд-мм-гггг — дата проведения измерений или формирования набора данных.

Имя файла должно использоваться как источник метаданных и отображаться в интерфейсе модуля, журнале обработки и имени выходного файла.

3. Структура входного CSV-файла

Входной CSV-файл должен:

иметь расширение .csv;
содержать строку заголовков;
использовать разделитель ;;
содержать следующие обязательные столбцы:
fuel
additive
component
Назначение столбцов:
fuel — значение параметра топлива;
additive — значение параметра разбавителя;
component — измеренная концентрация исследуемого компонента, полученная после газового анализа.

Все строки файла должны содержать корректные числовые значения в указанных столбцах.

4. Использование данных входного CSV

Данные из входного CSV используются следующим образом:

столбец fuel рассматривается как первая независимая переменная;
столбец additive рассматривается как вторая независимая переменная;
столбец component рассматривается как зависимая переменная, подлежащая аппроксимации.

На основе этих данных модуль строит аппроксимированную поверхность концентрации исследуемого компонента.

5. Требования к имени выходного файла

CSV-файл, сформированный после аппроксимации, должен сохраняться с именем следующего вида:

approx_наименование_топлива-разбавитель-компонент-дд-мм-гггг.csv

Пример:
approx_waste_oil-steam-CO-13-03-2026.csv

Префикс approx_ должен автоматически добавляться к имени исходного файла без изменения остальной части имени.

6. Структура выходного CSV-файла

Выходной CSV-файл должен:

иметь расширение .csv;
содержать строку заголовков;
содержать результаты аппроксимации в табличном виде;
включать следующие столбцы:
fuel
additive
component
Назначение столбцов выходного файла:
fuel — значение узла сетки по первой координате;
additive — значение узла сетки по второй координате;
component — аппроксимированное значение концентрации исследуемого компонента в данной точке.
7. Проверка корректности CSV-файлов

Перед запуском аппроксимации модуль должен проверять:

соответствие имени файла установленному шаблону;
наличие расширения .csv;
наличие заголовков;
наличие обязательных столбцов fuel, additive, component;
использование разделителя ;;
корректность значений в столбцах.

При нарушении хотя бы одного из требований модуль должен выдать пользователю сообщение об ошибке с указанием причины.

8. Дополнительные требования

Желательно предусмотреть:

отображение имени входного файла в интерфейсе;
автоматическое формирование имени выходного файла;
возможность сохранения результата в выбранную пользователем папку;
отображение в журнале обработки имени исходного и сформированного файла.

8. Обработка ошибок и исключительных ситуаций

Модуль должен корректно обрабатывать следующие ситуации:

входной файл не существует;
нет доступа к файлу;
файл не является CSV;
CSV имеет неверный разделитель или поврежден;
отсутствуют обязательные столбцы;
в обязательных столбцах содержатся некорректные данные;
параметры аппроксимации заданы неверно;
ошибка построения RBF;
ошибка применения фильтра;
ошибка сохранения результата;
отсутствие прав на запись в выходную директорию.

Для каждой ошибки должно выводиться:

краткое понятное описание;
имя файла, в котором возникла ошибка;
запись в лог.

В пакетном режиме ошибка одного файла не должна приводить к остановке всей обработки. Это соответствует текущей логике try/except в цикле обработки.

9. Нефункциональные требования
9.1. Производительность
Модуль должен обеспечивать приемлемое время обработки файлов стандартного объема.
Интерфейс не должен зависать на длительных операциях; вычисления желательно выполнять в отдельном рабочем потоке.
В пакетном режиме должен отображаться прогресс обработки.
9.2. Надежность
Модуль должен корректно обрабатывать ошибки без аварийного завершения всего GUI-приложения.
Все некорректные пользовательские действия должны обрабатываться через валидацию и сообщения интерфейса.
9.3. Расширяемость

Архитектура модуля должна позволять в будущем:

добавлять новые методы аппроксимации;
добавлять дополнительные фильтры;
добавлять визуализацию поверхности;
поддерживать другие форматы входных и выходных данных.
10. Требования к архитектуре модуля

Модуль GUI должен быть логически разделен как минимум на следующие части:

Слой интерфейса
Отвечает за взаимодействие с пользователем, выбор файлов, отображение статуса и логов.
Слой бизнес-логики
Управляет сценарием обработки: чтение → аппроксимация → фильтрация → экспорт.
Слой работы с данными
Использует существующие функции чтения, расчета, фильтрации и экспорта.

Такое разделение рекомендуется, поскольку текущая реализация уже естественным образом делится на reader, rbf, filtering, exporter, pipeline.

11. Требования к журналированию

Модуль должен вести журнал выполнения, содержащий:

дату и время запуска;
выбранные параметры;
список обрабатываемых файлов;
результаты обработки по каждому файлу;
тексты ошибок при их возникновении;
путь к сохраненным результатам.

Журнал должен отображаться в интерфейсе, а при необходимости — сохраняться в отдельный лог-файл.

12. Требования к тестированию

Должны быть предусмотрены проверки следующих сценариев:

Позитивные сценарии
загрузка корректного CSV и успешная обработка;
пакетная обработка нескольких корректных CSV;
обработка при разных значениях resolution;
обработка при включенном и выключенном clamp_zero;
обработка при разных значениях median_size;
успешное сохранение результатов.
Негативные сценарии
файл без одного или нескольких обязательных столбцов;
пустой CSV;
CSV с неверным разделителем;
нечисловые значения в ключевых полях;
недоступная выходная папка;
недопустимые значения параметров;
ошибка вычислительного ядра аппроксимации.
13. Требования к документации

Для модуля должна быть подготовлена пользовательская документация, включающая:

назначение модуля;
описание входного формата CSV;
описание параметров;
инструкцию по запуску одиночной и пакетной обработки;
описание выходного формата;
перечень возможных ошибок и способов их устранения.
14. Предлагаемый состав параметров по умолчанию

На основании текущей реализации рекомендуется использовать значения по умолчанию:

resolution = (100, 100)
median_size = 20
clamp_zero = True
kernel = "linear".
15. Критерии приемки

Модуль считается реализованным, если:

пользователь может выбрать файл или папку с CSV;
система корректно проверяет структуру входных данных;
выполняется аппроксимация поверхности по данным;
выполняется фильтрация по заданным параметрам;
результаты сохраняются в CSV в требуемом формате;
ошибки отображаются корректно и не приводят к сбою GUI;
пакетная обработка работает для набора файлов;
интерфейс позволяет задать все ключевые параметры обработки.
